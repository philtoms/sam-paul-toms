# Dev Audio Files Research

**Task:** KB-012
**Date:** 2026-05-19
**Status:** Research & Recommendations

## Problem

During local development, the audio player requires audio files served from a URL. In production, these come from Cloudflare R2 via a public URL. Locally, the default `R2_PUBLIC_URL=http://localhost:8788/r2` expects Wrangler's R2 dev proxy to be running — which requires bucket setup and actual audio files in R2.

Developers need a way to work on the audio player UI and logic without setting up R2.

## How Audio URLs Work (Current Flow)

1. **Frontmatter** defines relative paths: `audioFile: releases/echoes-ep/01-reverberations.mp3`
2. **`resolveAudioUrl()`** (`src/scripts/audio-helpers.ts`) prepends `R2_PUBLIC_URL`:
   ```
   http://localhost:8788/r2/releases/echoes-ep/01-reverberations.mp3
   ```
3. **`buildSources()`** (`src/components/AudioPlayer/audioEngine.ts`) generates OGG + MP3 fallback:
   ```
   [
     "http://localhost:8788/r2/releases/echoes-ep/01-reverberations.ogg",
     "http://localhost:8788/r2/releases/echoes-ep/01-reverberations.mp3"
   ]
   ```
4. **Howler.js** tries OGG first, falls back to MP3

### Key Constraints

- The OGG URL is auto-generated by replacing `.mp3` → `.ogg` in the resolved URL
- If only MP3 is available, Howler.js will simply skip the OGG source and play MP3 (graceful fallback)
- `resolveAudioUrl()` already supports absolute URLs — if `audioFile` starts with `http://` or `https://`, it's returned as-is
- CORS headers must be present for browser-side audio loading via Howler.js

## Option 1: Static Files in Astro's `public/` Directory ⭐ Recommended

### How It Works

Astro serves everything in `public/` at the root path during `astro dev`. If you place audio files in `public/audio-samples/`, they become available at:

```
http://localhost:4321/audio-samples/releases/echoes-ep/01-reverberations.mp3
```

### Setup

1. **Create directory structure:**
   ```bash
   mkdir -p public/audio-samples/releases/echoes-ep
   ```

2. **Add sample MP3 files** (small clips, ~30 seconds each, ~500KB each):
   ```
   public/audio-samples/releases/echoes-ep/01-reverberations.mp3
   public/audio-samples/releases/echoes-ep/02-hollow.mp3
   public/audio-samples/releases/echoes-ep/03-reflections.mp3
   public/audio-samples/releases/echoes-ep/04-resonance.mp3
   ```

3. **Set `.env`:**
   ```
   R2_PUBLIC_URL=http://localhost:4321/audio-samples
   ```

### Example Dev Audio URL

For a track with `audioFile: releases/echoes-ep/01-reverberations.mp3`:

```
http://localhost:4321/audio-samples/releases/echoes-ep/01-reverberations.mp3
```

OGG fallback URL (will 404, Howler falls back to MP3):
```
http://localhost:4321/audio-samples/releases/echoes-ep/01-reverberations.ogg
```

### Pros

| Factor | Assessment |
|--------|-----------|
| **Setup complexity** | Trivial — mkdir + copy files |
| **No external service** | ✅ Works offline |
| **No CORS issues** | ✅ Same origin as dev server |
| **Format support** | Any format Astro can serve (MP3, OGG, WAV) |
| **Code changes needed** | ❌ None — just change `R2_PUBLIC_URL` in `.env` |
| **Latency** | Instant (local filesystem) |
| **Reliability** | Excellent — no network dependency |
| **.gitignore** | Add `public/audio-samples/` to avoid committing large binaries |

### Cons

- Audio binary files must be manually obtained and placed (not version-controlled)
- Developers need to obtain sample MP3 files on their own
- Port must match Astro dev server port (default: 4321)

### Getting Sample Audio Files

Free sources for short sample MP3 files:

- **[Pixabay Music](https://pixabay.com/music/)** — Free CC0 music tracks, downloadable as MP3. Search for "ambient electronic" for genre-appropriate samples.
- **[Free Music Archive](https://freemusicarchive.org/)** — Large library of CC-licensed music. Filter by license for royalty-free options.
- **[Audacity](https://www.audacityteam.org/)** — Generate sine wave / white noise test tones locally. Export as MP3 via `File → Export → Export as MP3`.
- **[ffmpeg](https://ffmpeg.org/)** — Generate test audio programmatically:
  ```bash
  # Generate a 30-second 440Hz sine wave test tone
  ffmpeg -f lavfi -i "sine=frequency=440:duration=30" -b:a 128k public/audio-samples/releases/echoes-ep/01-reverberations.mp3
  ```

## Option 2: Python `http.server` (or any static file server)

### How It Works

Run a lightweight HTTP server serving a directory of sample audio files on a chosen port.

### Setup

1. **Create audio directory:**
   ```bash
   mkdir -p ~/dev-audio/releases/echoes-ep
   # Copy sample MP3 files into this directory
   ```

2. **Start server:**
   ```bash
   cd ~/dev-audio
   python3 -m http.server 8888
   ```

3. **Set `.env`:**
   ```
   R2_PUBLIC_URL=http://localhost:8888
   ```

### Example Dev Audio URL

```
http://localhost:8888/releases/echoes-ep/01-reverberations.mp3
```

### Pros

| Factor | Assessment |
|--------|-----------|
| **Setup complexity** | Low — one command |
| **No external service** | ✅ Works offline |
| **CORS** | ⚠️ Python's http.server doesn't send CORS headers by default. May need a wrapper script. |
| **Code changes needed** | ❌ None |
| **Separation** | ✅ Audio files completely separate from Astro project |

### Cons

- **CORS issue:** Python's `http.server` does not include `Access-Control-Allow-Origin` headers. Howler.js loads audio via `HTMLMediaElement` which **should** work without CORS for simple playback, but programmatic access (e.g., Web Audio API analysis) will fail. For basic playback testing this should work, but it's not a perfect simulation.
- Extra terminal window required
- Must remember to start/stop the server

### CORS-Fix Variant

Create a small Python server script with CORS headers:

```python
#!/usr/bin/env python3
"""Simple HTTP server with CORS headers for dev audio files."""
import http.server
import functools

class CORSHandler(http.server.SimpleHTTPRequestHandler):
    def end_headers(self):
        self.send_header('Access-Control-Allow-Origin', '*')
        super().end_headers()

http.server.test(HandlerClass=CORSHandler, port=8888)
```

Save as `scripts/dev-audio-server.py` and run with `python3 scripts/dev-audio-server.py`.

## Option 3: Free CDN-Hosted Sample Audio

### How It Works

Host sample audio files on a free external service and point `R2_PUBLIC_URL` at it.

### Viable Services

#### 3a. GitHub Releases (via raw.githubusercontent.com)

1. Create a separate repo (e.g., `sam-dev-audio`) or use GitHub Releases
2. Upload sample MP3 files as release assets
3. Use the raw/download URL as `R2_PUBLIC_URL`

**URL pattern:**
```
https://raw.githubusercontent.com/<user>/sam-dev-audio/main/audio/releases/echoes-ep/01-reverberations.mp3
```

| Factor | Assessment |
|--------|-----------|
| **Setup complexity** | Medium — create repo, upload files |
| **CORS** | ⚠️ `raw.githubusercontent.com` sends `Access-Control-Allow-Origin: *` ✅ |
| **File size limit** | 100MB per file via web, 2GB via git-lfs |
| **Rate limit** | 60 requests/hour unauthenticated (more than enough for dev) |
| **Code changes needed** | ❌ None |
| **Offline** | ❌ Requires internet |

#### 3b. Cloudflare R2 Free Tier (Actual R2, but separate dev bucket)

1. Create a free Cloudflare R2 bucket for dev audio
2. Upload sample files
3. Enable public access URL
4. Point `R2_PUBLIC_URL` at the dev bucket's public URL

**URL pattern:**
```
https://pub-<dev-bucket-id>.r2.dev/releases/echoes-ep/01-reverberations.mp3
```

| Factor | Assessment |
|--------|-----------|
| **Setup complexity** | Medium — Cloudflare dashboard setup |
| **CORS** | ✅ Fully configurable |
| **Free tier** | 10GB storage, 10M reads/month |
| **Code changes needed** | ❌ None |
| **Offline** | ❌ Requires internet |
| **Fidelity** | ✅ Identical to production setup |

#### 3c. Internet Archive

1. Upload a small "item" with sample audio files
2. Use the download URLs

**URL pattern:**
```
https://archive.org/download/sam-dev-audio/releases/echoes-ep/01-reverberations.mp3
```

| Factor | Assessment |
|--------|-----------|
| **Setup complexity** | Medium — account + upload |
| **CORS** | ⚠️ Internet Archive sends permissive CORS headers ✅ |
| **Rate limit** | Reasonable for dev use |
| **Code changes needed** | ❌ None |
| **Offline** | ❌ Requires internet |
| **Permanence** | ⚠️ IA may flag inactive items |

## Option 4: Use Absolute URLs in Frontmatter (Development Override)

### How It Works

Instead of changing `R2_PUBLIC_URL`, override specific track `audioFile` fields with absolute URLs pointing to publicly available sample audio.

### Example

Any public MP3 file URL can be used directly. Since `resolveAudioUrl()` returns absolute URLs unchanged, you could temporarily change frontmatter:

```yaml
tracks:
  - title: Reverberations
    audioFile: https://www.soundhelix.com/examples/mp3/SoundHelix-Song-1.mp3
```

### Pros

- Zero infrastructure needed
- Works with any public MP3 URL
- Tests the exact code path used in production (no env var override)

### Cons

- Modifies content files (frontmatter), which is messy and easy to accidentally commit
- Depends on external URL availability
- Not a scalable approach for multiple tracks/releases
- **Not recommended as primary approach** — better as a quick one-off debugging tool

## Recommendation

### Primary: Option 1 — Astro `public/` directory

**Best for most developers.** Zero infrastructure, works offline, no CORS issues, and requires no code changes. The only manual step is obtaining sample MP3 files.

### Setup Steps

1. **Add to `.gitignore`:**
   ```
   # Dev audio samples (large binary files)
   public/audio-samples/
   ```

2. **Create a setup script** (optional, for convenience):
   ```bash
   #!/bin/bash
   # scripts/setup-dev-audio.sh
   # Generates placeholder audio files for local development

   mkdir -p public/audio-samples/releases/echoes-ep

   tracks=("01-reverberations" "02-hollow" "03-reflections" "04-resonance")

   for track in "${tracks[@]}"; do
     if [ ! -f "public/audio-samples/releases/echoes-ep/${track}.mp3" ]; then
       echo "Generating ${track}.mp3..."
       ffmpeg -f lavfi -i "sine=frequency=$((400 + RANDOM % 400)):duration=30" \
         -b:a 128k "public/audio-samples/releases/echoes-ep/${track}.mp3" \
         -y 2>/dev/null
     fi
   done

   echo "Dev audio files ready."
   echo "Set in .env: R2_PUBLIC_URL=http://localhost:4321/audio-samples"
   ```

3. **Set `.env`:**
   ```
   R2_PUBLIC_URL=http://localhost:4321/audio-samples
   ```

4. **Run the setup script and start dev:**
   ```bash
   bash scripts/setup-dev-audio.sh
   npm run dev
   ```

### Configuration Summary

| Variable | Production | Development |
|----------|-----------|-------------|
| `R2_PUBLIC_URL` | `https://pub-xxxx.r2.dev` | `http://localhost:4321/audio-samples` |

### No Code Changes Required

The existing `resolveAudioUrl()` and `R2_PUBLIC_URL` pattern is already flexible enough. Simply overriding the env variable routes all audio through the local Astro dev server's static file serving. No changes to `audio-helpers.ts`, `audioEngine.ts`, or any other source files are needed.

### Limitations

1. **OGG fallback will 404** — Only MP3 files would be generated; the OGG URL will return 404. This is fine because Howler.js falls back gracefully from OGG to MP3. The player will work correctly.

2. **Astro dev server port** — Default is 4321. If a developer changes the port via `astro.config.mjs` or CLI flags, `R2_PUBLIC_URL` must match.

3. **Audio content is placeholder** — Generated sine waves won't sound like real music, but they exercise the full audio pipeline (load → decode → play → seek → time tracking → track advancement).

4. **Not suitable for testing streaming behavior** — Local files load instantly; they don't simulate the progressive loading of remote R2 files. For that, Option 3b (R2 dev bucket) would be more appropriate.
